[`0:02`](https://youtu.be/AmdLVWMdjOk?t=2) 模型发展日新月异。若在三个月或六个月后问我这个问题，答案将截然不同。这里是 Boris Chney——Claude Code 的创建者，前 Meta 首席工程师。我们探讨了塑造其职业生涯的方方面面。能否请您解释下“潜在需求”？

[`0:26`](https://youtu.be/AmdLVWMdjOk?t=26) 我的天，说“困难”都太轻描淡写了，简直是一场噩梦。我们还聊到了Claude Code，以及Anthropic目前实际在推进的事情。尽管Anthropic的规模增长了两倍，但得益于Claude Code，每位工程师的生产力提升了近70%。不要为今天的模型而构建，要为六个月后的模型而构建。我会推荐给每个人的一本技术书籍，也是对我作为工程师影响最大的一本，是……对于与OpenAI的Codeex的竞争，你有什么看法？我想从你故事的起点开始——你在Meta晋升为高级工程师的时候。是什么样的项目让你获得了晋升，当时你的情况是怎样的？如果我没记错的话，那个项目是关于“聊天与群组”的，目的是让Messenger和Facebook更紧密地结合。实际上，我在Meta最初参与的几个项目都与Messenger和Facebook有关。我认为第一个是Zach提出的一个想法，关于同步Messenger聊天记录和Facebook群组。嗯，当时有好几个项目，都是试图把Messenger和Facebook拉得更近一些。我们的动机是感觉到，这种公共空间式的社交产品正在逐渐消失，而事物正更多地转向聊天以及更随意的实时交流空间。因此，我们尝试了产品的几个版本，“聊天与群组”是那个成功的方案。

我觉得当时它大概排在第三或第四位的样子，我当时在Facebook群组里活跃，或者说我那时主要在Facebook工作，很多时间都花在Messenger上——从组织架构来说这两块其实离得很远。这是Steve（当时是PM/产品经理）提出的一个构想，觉得我们应该做这个功能，我立刻就接住了这个想法，心想“太棒了我们就该干这个”。于是我开始动手捣鼓起来。很快出现了些积极的迹象，我就申请了更多工程师支持。后来加入了三位工程师：尚塔布丽、Crystal和常。他们是这个项目最早的三位工程师，接着我们获得了一些数据科学和设计支持。最初只在网页端开发，后来也慢慢拓展到移动端。总的来说，我们验证了在Facebook群组内嵌聊天功能是可行的，这类产品确实能运作起来。不过说实话，很多方面都没能成功，以现代产品标准来看，当时的体验简直粗糙得离谱。那个年代大家都在网页端开发，各种bug完全被容忍。如今我觉得标准真的提高了很多，无论是视觉呈现还是质量要求都严格多了。随着产品成长，我们团队规模很小，每个人都得身兼数职。我记得当时没有用户研究员，我就趁午休时间去公司餐厅，拿着新功能给餐厅工作人员演示，问他们：“你能找到怎么打开聊天窗口吗？”有时候他们能找到，有时候找不到——这就是最原始的观察式用户研究。通过观察特定情境下人们如何完成任务，且不过多提示，来获得真实反馈。

所以你不会透露太多，而是观察他们在哪里遇到困难。看看他们理解了多少。嗯，我们就这样做，然后我也教团队怎么操作。很快，我们所有人午餐时都会去自助餐厅，开始“骚扰”餐厅工作人员——就把他们当作典型用户，问：“这样表达清楚吗？有没有道理？” 

有趣的是，早期Facebook的文化让工程师能做代码之外这么多事情。比如你在做用户研究（UXR），我记得在你的故事里你还参与过设计工作，并且指导别人做设计。我认为这是Facebook文化中一个相当独特且有趣的特点。

我觉得这一点非常重要，时至今日，在我所在的quad code团队，我们非常重视多面手。我喜欢和多面手合作。如果你是工程师，会写代码，但也能做产品工作，也能做设计，有产品意识，愿意主动去和用户交流——我特别喜欢这类工程师。实际上，我们现在招聘所有职能岗位都是这个思路。比如我们的产品经理会写代码，数据科学家会写代码，嗯，我们的用户研究员也会写一点代码。我就是喜欢这样的多面手。我觉得这其实是我从职业生涯初期就形成的习惯——从18岁第一次创业开始，我就得事事亲力亲为，直到去Facebook之前，我都在小型公司工作，什么都得做。某种程度上，在大公司会被限制在特定职责范围内，但那种界限其实很模糊，毕竟，“工程师”到底是什么呢？

就像它是一套非常窄的技能组合，但你实际做的事情是打造产品或者搭建基础设施，而要端到端地完成这些事情，除了写代码之外还有太多需要考虑的因素。能在一个地方——我觉得Facebook在当时确实独特地奖励了那种能力——真的很棒。而且我记得在那个半年周期结束时我得到了晋升，然后在下个半年周期里，团队里每位工程师也都获得了晋升。在那些早期产品中，有个概念叫"潜在需求"，你提过好几次，听起来它是推动很多产品方向发展的关键因素。你能解释一下什么是"潜在需求"吗？

[`5:42`](https://youtu.be/AmdLVWMdjOk?t=342) 潜在需求我认为是产品设计中最核心的原则。观察Facebook那些成功的产品，每一款都蕴含着潜在需求的要素。以marketplace为例，它源于当时对Facebook小组的观察——发现40%的帖子都是在买卖商品。Facebook小组原本并非为商业交易设计，但用户却自发地将它用于交易。这其实挺有意思的：你设计的产品允许用户变相使用，甚至存在一些被滥用的空间，接着你通过数据观察用户如何"滥用"它，然后围绕这种行为模式开发新产品。从Facebook小组到买卖小组，再到marketplace，这个过程很自然地延续了用户既有的意图——毕竟大家早就想在Facebook小组里进行商品交易了。我认为Facebook Dating的逻辑也很相似。当时的观察显示约60%的个人主页浏览行为发生在未互加好友的异性之间。这种传统意义上的"悄悄关注"行为，对于当时的团队如Nate和FMS来说，恰恰证明了社交约会功能的可行性。这个产品原则的核心在于：你永远无法让用户去做他们从未尝试过的事情。

你可以做的是，找到他们的真实意图，然后引导它，让他们更好地利用这个意图，嗯，更轻松地实现他们想要的事情。我也觉得在你故事的这个部分，你提到过你跨组织工作。你当时是因为要弥合Messenger和很多工程小组之间的工作差距而进行跨部门协作的。我很好奇，你说当时存在明显的文化差异，而且这很困难。呃，对于在文化差异很大的组织间工作，你有什么建议吗？

[`7:30`](https://youtu.be/AmdLVWMdjOk?t=450) 天哪，“困难”这词简直太轻描淡写了。那简直是场噩梦。当时在Facebook，我们只想着尽可能快速地行动，尽快推出出色的产品。而Messenger团队唯一关心的就是可靠性和性能。我们的价值观简直是背道而驰。

这不仅仅是文化差异，不只是工程师之间的隔阂。那个团队的工程师对我们充满戒心，因为我们的工作会影响他们的性能指标。在组织架构上，他们的目标是缓慢迭代而不让指标下降，而我们则被要求快速上线。两队的目标完全不同——他们关心服务等级协议和正常运行时间，而我们只在乎日活跃用户数和用户参与度。

我从中领悟到，这类文化价值观的影响极其深远。它不只是口头说说，你能从组织设计、目标设定乃至方方面面的细节中切实感受到它。

说实话，我认为那个项目（最终确实演变成了成功的产物，但当时那个版本）失败的原因之一，正是这种价值观的根本差异。所以我想，如果要让价值观迥异的团队真正协同并取得成功，就必须找到某种共同的目标、共同的利益、共同的信念，或者某个双方都愿意共同验证的假设——一个若成功便会令双方都深感振奋的方向。

嗯，我觉得这种聊天和群组功能对Facebook来说确实很酷，但对Messenger来说，由于很多原因，就没那么有趣了。所以，如果你现在回顾当时的情况，会怎么调整那个项目的做法呢？

[`9:10`](https://youtu.be/AmdLVWMdjOk?t=550) 我觉得我可能会直接去找扎克伯格说，如果你真的重视这件事，我们就应该把 Messenger 整合到 Facebook 里。不过我记得后来这确实发生了，而且实际上发生了好几次：Messenger 一开始在内部，然后被分出去，又整合回来，再分出去。大公司就是这样。但从根本上说，要让这类事情成功，共同负责人不能是像 Chris Cox 那样的高管，应该由层级稍低的人来负责。这样组织架构才能更强调协作。

[`9:39`](https://youtu.be/AmdLVWMdjOk?t=579) 明白了，这样能调整激励机制，避免持续的内部角力。

[`9:44`](https://youtu.be/AmdLVWMdjOk?t=584) 对，正是这样。在你这个职业生涯阶段，我注意到你做过很多有趣的副业项目，我很好奇这类项目会产生怎样的蝴蝶效应？比如，甚至在进入 Meta 之前，你就开发了 Undux——React 的状态管理框架。这对你职业生涯有什么影响吗？

[`10:07`](https://youtu.be/AmdLVWMdjOk?t=607) 对我来说，副业探索特别重要。招工程师时，我一定会关注这点。我希望看到候选人有各种副业探索：有意思的周末项目、有趣的业余项目，甚至只是特别热衷于酿制康普茶之类的人。我们需要的是对主业之外的事物仍保持好奇和兴趣的人。这类人通常更全面发展，也是我最喜欢共事的类型。我认为我的许多成长正来源于此类副业项目。

比如 Undux 的起源，其实是觉得 React 状态管理有点复杂过头了。当时主流方案先是 Flux，后来是 Redux，但我怎么也想不明白 Redux。我得说，我算是个普通工程师，主要做产品开发，不是那种顶尖的系统架构师。当时 Redux 的 reducers 这些概念，还有更新小状态都要走复杂流程的设计，让我特别困惑。

所以我做了个更简单的方案，用起来效果不错。当时我在非营利组织做志愿者，他们也开始用这个方案，工程师们反响很好。加入 Facebook 后，我发现大家对 Redux 的使用普遍感到沮丧——公司内部有个 Redux 用户群组，很多问题都和我的困惑一样。

作为工程师或产品人员，当你遇到某个问题时，有时可能只是个人困惑，但往往其他人也会遇到。培养判断“这个问题是否普遍”的直觉很重要，而当时明显是个普遍问题。从技术支持帖子和我的团队使用 Redux 的困难中都能看出这点。

于是我在公司内部推出了 Undux。这产品其实也就“还行吧”，算不上多优秀，但至少比 Redux 好用些。当时我不太清楚怎么推广，就先发了帖子。渐渐有人开始使用，记得通知组的 Jeff Case 是早期重度用户，我们还一起熬夜调试过几个棘手的通知相关漏洞。

为了扩大推广，我写了个小脚本抓取问题报告群组的数据，按团队统计问题数量。然后通过聊天工具直接联系每个团队的技术主管和经理，为每个团队安排专门的技术分享。几个星期里，我大概做了二三十场，甚至可能有四十场这样的分享会。
[`12:45`](https://youtu.be/AmdLVWMdjOk?t=765) 我记得当时就在Meta园区里骑着自行车做这类分享，真的很有意思，因为听众们都非常投入，他们真的很兴奋，因为终于有人关心解决这个他们真正遇到的问题。我觉得在某个阶段，它差点成了Facebook上最流行的状态管理框架，但后来很快就被Recoil这类更现代的方案取代了，现在则是Relay那些了。这类副项目会体现在你的绩效评估里吗？或者说在某些方面帮到你？嗯，我觉得它算是写进绩效评估了。但按照Meta的标准，这更像是锦上添花，本身并不是能让你晋升的那种决定性因素。不过那时候我也做了不少其他“支线任务”。比如有一阵子我特别迷TypeScript——其实是在前一家公司的时候。当时我们用这门语言，但好的学习资料很少。所以我就想，得有人来写本书介绍它，这事太不可思议了居然没人做。这语言设计得实在太棒了，那些理念在当时其他语言里根本没有。像后来才有的条件类型、字面量类型、映射类型……这些特性简直疯狂到连最硬核的Haskell程序员都会佩服，但当时根本没人系统写这些。我就一头扎进去写那本书，结果它几乎吞掉了我一年的生命。[笑] 真不建议学我。不过深入钻研的感觉确实很爽。后来我还在旧金山搞了可能是当时最大的TypeScript技术聚会，特别酷的是能遇到Ryan Dahl（Node.js创始人）这些JavaScript界的名人，让我意识到他们也都是普通人——大家都会做出有趣的东西，有些在特定时期很酷，但说到底都是人在创造，谁都可以尝试。

[`14:47`](https://youtu.be/AmdLVWMdjOk?t=887) 后来你在Meta，或者甚至在Anthropic时，用到TypeScript或那些技术积累了吗？嗯，说来有趣。我以前其实不太在乎编程语言，直到大概十年前——当时我骑摩托车出了挺严重的事故，摔断了胳膊。

[`15:08`](https://youtu.be/AmdLVWMdjOk?t=908) 老天，那你当时怎么写代码的？

[`15:10`](https://youtu.be/AmdLVWMdjOk?t=910) 呃，所以这是最难的部分。我差不多有一个月完全没法编程，而且我的手还是会疼，所以我没办法写JavaScript——那是我当时主要用的语言。实际上我不得不去拓展学习其他语言，因为它们需要的按键次数更少。于是我先尝试了CoffeeScript，因为它的括号之类的东西比较少。不过我觉得这个语言现在基本没人用了。但这也是我接触Haskell和函数式编程的契机，因为你可以用更少的按键实现同样的功能，这成了当时最直接的动力。后来我在一家对冲基金工作——那还是在Facebook出现之前——我有个同事叫Rick，他特别喜欢Scala，而我之前完全不理解这个语言。是他真正带我入门，让我接触到函数式编程的领域。直到现在，如果让我推荐一本对工程师影响力最大的技术书，我还是会推荐这本《Scala中的函数式编程》。虽然你今天可能完全用不上Scala，但它教会你思考编程问题的方式，完全颠覆了大多数人通过实践或学校教育形成的编码思维。它真的会彻底改变你的编程方式。对我来说，这个学习路径大概是：先是Haskell和CoffeeScript这类语言作为入门，然后是Scala，接着是TypeScript——我觉得这彻底改变了我的思维方式，因为现在我写代码时会用类型思考。代码中最重要的其实是类型签名，这比代码本身更重要。把这部分处理好，代码就会非常简洁。即使在Facebook期间我主要写Flow和Hack，后来在Instagram写Python，这种思维方式都非常有帮助。现在在Anthropic我主要写TypeScript和Python，所以这些经验依然很实用。但我认为更重要的启示是：要学会用类型来思考。
[`17:07`](https://youtu.be/AmdLVWMdjOk?t=1027) 在你职业生涯的这个阶段，你提到过你入职时是降级进来的，尽管经验丰富却只被定为中级工程师。你说事后想想，很庆幸当时是降级入职的。我很好奇，这背后的考量是什么？

[`53:21`](https://youtu.be/AmdLVWMdjOk?t=3201) “下线功能”的理念是：如果只是不断给应用添加新功能，这对少数用户来说可能很棒，但实际上对大多数不使用该功能的用户是有害的。你可以想象一个应用，我们只往里面加功能。随着时间的推移，功能越积越多。如果每个功能只有大约10%的人在使用，那么普通用户就会看到一堆他们用不上的功能。这样一来，应用界面就显得杂乱又令人困惑。当他们打开应用时，会不知所措。而且，从根本上讲，屏幕尺寸是有限的。这是宝贵的“不动产”，是一个有限的资源，所有不同的功能都在争夺它。所以，添加一个新功能，实际上就挤占了另一个用户本可能用到的功能的机会。因此，“下线功能”这个理念就是：一个功能必须达到某种使用率门槛，如果达不到，我们就直接删掉它。一小部分用户可能会因此不满，但这实际上对大多数用户来说是件好事。平均来看，这对所有人都是真正有益的。
